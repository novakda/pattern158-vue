### tsconfig constraint — why the DOM lib reference is mandatory

`tsconfig.editorial.json` declares `"lib": ["ES2022"]` only (no DOM). That's intentional — editorial scripts run in Node, not a browser. But convert.ts references DOM types (`Document`, `Element`, `HTMLImageElement`) in function signatures and body casts. Without a file-scoped `/// <reference lib="dom" />` directive at the top of convert.ts, TypeScript will emit `Cannot find name 'Document'` errors during `pnpm build`.

**The fix** — mirror `scripts/editorial/capture.ts` line 13 exactly: put the triple-slash directive at the top of the file (above `import` statements). capture.ts uses this pattern because Playwright `page.$$eval` / `page.waitForFunction` callbacks run in the browser context and reference `document` + DOM types; convert.ts has the same need for different reasons (its happy-dom `Window.document` is typed with standard DOM interfaces).

```ts
// scripts/editorial/convert.ts (top of file, before imports)
/// <reference lib="dom" />
```

### DOM parser — happy-dom (already a devDep)

**Preferred path:** use `happy-dom` (package.json line 39: `"happy-dom": "^20.8.4"` in devDependencies). It's a top-level install, hoisted by pnpm, deterministic (no wall clock, no randomness), and exposes a standards-compliant DOM. happy-dom is how the project's Vitest DOM tests already run — the codebase treats it as the canonical Node-side DOM.

happy-dom construction shape:
```ts
import { Window } from 'happy-dom'
const window = new Window()
const document = window.document
document.documentElement.innerHTML = `<html><body>${rawHtml}</body></html>`
// ... sanitize on document.body ...
const result = document.body.innerHTML
```

The `window.document` has a standards-compliant `querySelectorAll`, `createElement`, `replaceWith`, `attributes`, `removeAttribute`, `innerHTML` surface — every API this plan needs.

**Why NOT domino:** `@mixmark-io/domino` is a transitive dep of turndown, but pnpm strict hoisting means a direct `import from '@mixmark-io/domino'` from convert.ts will fail module resolution. We do not add it as a direct devDep (CONTEXT.md forbids "no extra dep added" — and happy-dom already satisfies the parser need).

**Why NOT Turndown's internal parser:** Turndown exposes `.turndown(input)` which accepts either an HTML string OR a DOM node, but does not publicly expose its internal parser as a standalone API. Using it indirectly is fragile.

**Locked decision:** use happy-dom. Document in comments, grep-assert in acceptance.

        * demote headings (h1→h3, clamp at h6), serialize body.innerHTML back.

        * CONV-02 (sanitization) + CONV-04 (heading demotion).

       export function sanitizeHtml(rawHtml: string): string {
         const window = new Window()
         const document = window.document
         document.documentElement.innerHTML = `<html><body>${rawHtml}</body></html>`
         // Step 2: strip subtrees.
         document.querySelectorAll('script, style, noscript, [aria-hidden="true"]').forEach((el) => el.remove())

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-49-01 | Tampering | sanitizeHtml — HTML injection into output markdown | mitigate | Turndown's default rules escape raw HTML fragments; this plan's sanitizer removes `<script>/<style>/<noscript>` BEFORE Turndown runs, so any injected `<script>` can never reach the Markdown output. Block-level element escaping is Turndown's responsibility downstream; we guarantee no script-carrying subtrees survive sanitization. |
| T-49-02 | Denial of Service | sanitizeHtml — pathological HTML (deep nesting, many elements) | accept | happy-dom has O(n) parse complexity. The largest realistic input is the FAQ page main-content HTML (~200KB, ~5K elements). `querySelectorAll('*')` is O(n), attribute iteration is amortized O(1) per attr. | T-49-03 | Tampering | demoteHeadings — prototype pollution via crafted heading content | mitigate | `oldEl.tagName.charAt(1)` returns a string char. `Number()` on non-digit char returns NaN; `Math.min(6, NaN + 2) === NaN`, which fails the `newLevel === currentLevel` guard loosely but `newLevel = NaN` passed to `createElement(`h${NaN}`)` throws a DOM error, surfacing malformed input as a loud failure rather than silent corruption. | T-49-05 | Information Disclosure | aria-hidden subtree leakage (hidden icons, decorative content) | mitigate | Subtree strip removes every `[aria-hidden="true"]` element and all children. Editorial corpus never contains the "visually hidden" content a screen reader would skip. |
