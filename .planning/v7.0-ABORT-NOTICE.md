# v7.0 ABORT NOTICE

**Date:** 2026-04-19
**Milestone aborted:** v7.0 Static Markdown Export Pipeline
**Progress at abort:** 2 of 9 phases shipped (Phases 037, 038), Phase 039 in planning only
**Replacement direction:** Playwright-based live-site editorial capture + subsequent TBD static rebuild

## Summary

v7.0 was designed as a multi-renderer content export pipeline: extract content from Vue SFCs into typed modules (Phase 037, shipped), build an IR and markdown primitives (Phase 038, shipped), then layer seven per-page extractors (Phase 039, planning done), a dynamic exhibit extractor (Phase 040), monolithic and Obsidian renderers (Phases 041-042), orchestrator and writer (Phase 043), build integration (Phase 044), and documentation (Phase 045).

The pipeline was well-engineered for its design goal: producing derived renderings (monolithic Markdown, Obsidian vault) from the canonical `src/content/*.ts` modules with type preservation, drift detection, and round-trip guarantees.

The pipeline is the wrong tool for the actual current need.

## Why the pivot

**The current need is editorial, not publication.**

Dan's immediate goal is to read the full rendered site as a single semantic document, make editorial decisions (consolidate, reorganize, rewrite), and use that edited content as the foundation for a rebuild — likely as static HTML first, with the Vue site continuing in parallel as one implementation in a longer-term Rosetta Stone portfolio project across multiple frameworks.

v7.0's architecture assumes the `src/content/*.ts` modules are the canonical source of truth. For an editorial pass they are not canonical enough:

1. **Incomplete coverage.** `src/content/*.ts` captures the prose Phase 037 extracted from SFCs, but data modules (`src/data/json/*.json` — specialties, stats, brandElements, findings, influences, techPills, exhibit data, etc.) are consumed directly by components and only appear as prose when rendered. Any hardcoded component text also only surfaces at render time.
2. **No reading-order fidelity.** Source modules are unordered object shapes. Page templates assemble them into the order a reader actually encounters. Editorial decisions need the assembled order, not the source order.
3. **No composition fidelity.** Component composition, conditional rendering, dynamic data interpolation, and navigation context only exist in rendered output. These matter for editorial judgment.
4. **Dynamic routes.** Exhibit detail pages and case file filtering only materialize their actual prose when a route is hit. Source files declare templates; only the render shows what a real exhibit page reads like.

The rendered live site is the highest-fidelity representation of the site's actual content. For editorial work, capture the truth, not the source.

**The pipeline also locks in a direction that may not be the right long-term choice.**

v7.0 treats Vue as the canonical source of truth and Markdown/Obsidian as derived artifacts. The longer-term Rosetta Stone vision inverts that: neutral content (Markdown or JSON), with Vue being *one* of several rendering implementations alongside static HTML, Svelte, 11ty, etc. Continuing v7.0 would accrue architectural commitments to a direction that may need to be reversed.

**The pipeline is solving problems that aren't present.**

The IR's type preservation, drift guards, and round-trip tests are genuine engineering value for a production pipeline with multiple downstream consumers. For a one-off editorial read, they are overhead. The simplest tool for the editorial job is live-site capture.

## What's retained

**Phase 037 output is kept.** The `src/content/*.ts` modules are still the right place for the prose they contain, regardless of whether v7.0 ships. They improved the Vue codebase independent of the pipeline.

**Phase 038 output is kept.** The IR primitives in `scripts/markdown-export/primitives/` (and related) are shipped, tested, and harmless to leave in place. They may be reused if a different pipeline direction is chosen later.

**Phase 039 planning docs are kept.** `.planning/phases/039-static-page-extractors/` documents a thoughtful design for per-page extractors. If a future pipeline direction resurfaces the need, the work isn't lost — it's a reference.

**All milestone audits, ROADMAP, and PROJECT.md entries are kept.** The history of the work is a record of thinking and shouldn't be erased.

## What's stopped

- No further Phase 039 execution
- No Phase 040-045 work
- No new v7.0 plans, phases, or milestones
- No further IR primitives or extractor development unless explicitly tied to a new milestone
