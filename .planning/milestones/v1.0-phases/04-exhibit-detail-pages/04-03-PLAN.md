Storybook routing note (from RESEARCH.md):
ExhibitDetailPage calls useRoute() to read route.params.slug. In Storybook, useRoute() returns an empty params object (no slug set) — exhibit computed value will be null, and router.replace({ name: 'not-found' }) will fire. To render actual content, the story must provide a router context with a slug param, OR the component must have a defensive fallback for stories.

Defensive fallback approach (lowest friction — no decorator needed):
In ExhibitDetailPage.vue, if exhibit is null and we are NOT in a test environment, fall back to exhibits[0] for display purposes. However, this changes production behavior.

Preferred approach: Use a Storybook decorator that wraps the story in a mock router providing a known slug. The @storybook/vue3-vite integration supports Vue Router via `vue-router/stub` or a custom decorator.

Story decorator approach:
```typescript
import { createRouter, createWebHashHistory } from 'vue-router'

const mockRouter = createRouter({
  history: createWebHashHistory(),
  routes: [
    { path: '/exhibits/:slug', component: ExhibitDetailPage },
    { path: '/:pathMatch(.*)*', name: 'not-found', component: { template: '<div>Not Found</div>' } },
  ],
})

// Force the router to the exhibit-a route before rendering
await mockRouter.push('/exhibits/exhibit-a')

const meta = {
  decorators: [
    () => ({
      template: '<story />',
      setup() { return {} },
    }),
  ],
  ...
}
```

Simpler alternative: Pass the router as a global plugin in story args:
```typescript
export const Default: Story = {
  decorators: [
    (story) => ({
      components: { story },
      setup() {
        const router = createRouter({
          history: createWebHashHistory(),
          routes: [
            { path: '/exhibits/:slug', component: ExhibitDetailPage },
            { path: '/:pathMatch(.*)*', name: 'not-found', component: { template: '<div />' } },
          ],
        })
        router.push('/exhibits/exhibit-a')
        return { router }
      },
      template: '<story :router="router" />',
    }),
  ],
}
```

The RESEARCH.md notes this as LOW confidence — executor should verify which approach works with @storybook/vue3-vite and vue-router. If both fail, the acceptable fallback is: add an optional `exhibitOverride` prop to ExhibitDetailPage that bypasses the slug lookup (used only in stories), then update src/pages/ExhibitDetailPage.vue to accept it.
