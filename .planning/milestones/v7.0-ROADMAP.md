## v7.0 Static Markdown Export Pipeline

**Started:** 2026-04-10
**Aborted:** 2026-04-19

**Milestone Goal:** Generate two static markdown artifacts alongside the site build — a single monolithic document mirroring the site tree by heading levels, and an Obsidian-ready vault folder with one markdown page per site page organized to match the menu structure.

**Outcome:** Aborted in favor of a Playwright-based editorial capture approach (v8.0). The pipeline was well-engineered for producing derived renderings from `src/content/*.ts` modules, but the actual current need is editorial capture of the rendered live site. See `.planning/v7.0-ABORT-NOTICE.md` for full rationale.

### Key Scoping Decisions (preserved for reference)

- Content sourcing: Option B — refactor SFCs to import prose from `src/content/*.ts`, Phase 1 prerequisite (SHIPPED as Phase 37)
- Script runtime: standalone `tsx` script, NOT a Vite plugin
- Renderer architecture: two-mode renderer sharing a `DocNode` IR
- Dependencies: 3 new devDeps — `tsx`, `yaml`, `github-slugger` (all added in Phase 38)
- Tag format: flat kebab-case
- FAQ granularity: one note per page, questions as H3 + block anchors
- Monolithic depth: full exhibit details inline
- Imports: relative only inside `scripts/markdown-export/**`
- Testing: snapshot testing at small granularity + two-run determinism test
- CI drift guard: `npm run build:markdown && git diff --exit-code docs/`
