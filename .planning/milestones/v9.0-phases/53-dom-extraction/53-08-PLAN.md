Algorithm (deep-walk, heading-anchored segments):
1. Find `<main id="main-content">` OR fall back to the body element if absent (caller may pass pre-scoped mainHtml).
2. Flatten all h1..h6 in DOM document order — populate `headings`.
3. `title` = first h1 text, or `''`.
4. Build `segments`:
   - Walk the same flat heading list: for each heading H, find the textContent "owned" by H.
   - Ownership rule: H owns all non-heading text from H onward until the NEXT heading in document order (any level). "Non-heading text" = concatenated textContent of intermediate descendants excluding other h1..h6 elements and their subtrees.
   - Emit `{ heading: H, text }` per heading.
5. Skip empty-text segments from output (the hero h1 with only a subtitle `<p>` still emits a segment if subtitle has content; only truly-empty segments skipped).

Simpler implementation strategy (LOCK): walk the main element tree with a TreeWalker over text + heading elements:
- On encountering any h1..h6: close the current segment, open a new one anchored to this heading.
- On encountering any non-heading text node: append its textContent to the current segment's buffer.
- At end of walk: close the final segment.

Trees where a heading appears nested inside another element (e.g. `<li><h3>...</h3></li>`) still work because TreeWalker is depth-first document-order.

Because TreeWalker on happy-dom is supported (lib.DOM reference), use `document.createTreeWalker(mainEl, NodeFilter.SHOW_ELEMENT | NodeFilter.SHOW_TEXT)`.

If TreeWalker proves troublesome under happy-dom (check during implementation), a fallback: iterate `mainEl.querySelectorAll('*')` and track current heading + accumulate textContent per element that is NOT an h1..h6 AND whose closest ancestor heading is the current one. Favor the TreeWalker approach first.

```typescript
// scripts/tiddlywiki/extractors/pages.ts
// EXTR-07 — non-exhibit page-content extractor.
// SCAF-08: no setTimeout, no Date.now(), no new Date(), no Promise.all.

import {
  parseHtml,
  type PageContent,
  type PageHeading,
  type PageSegment,
} from './types.ts'

const HEADING_SELECTOR = 'h1, h2, h3, h4, h5, h6'
const HEADING_TAGS = new Set(['H1', 'H2', 'H3', 'H4', 'H5', 'H6'])

function textOf(el: Element | null): string {
  return (el?.textContent ?? '').trim()
}

function levelFromTag(tagName: string): number {
  return Number(tagName.charAt(1))
}

function collectSegmentText(
  headingEl: Element,
  nextHeadingEl: Element | null,
): string {
  // Walk forward in document order from headingEl, collecting text from every
  // element until (but not including) nextHeadingEl. We use a TreeWalker over
  // text nodes starting AFTER headingEl.
  const root = headingEl.ownerDocument.body
  const walker = root.ownerDocument.createTreeWalker(
    root,
    0x5, // SHOW_ELEMENT (1) | SHOW_TEXT (4) — literal value; NodeFilter.SHOW_* constants may not exist on happy-dom
  )
  const parts: string[] = []
  let started = false
  let skipHeading = false
  let node: Node | null = walker.firstChild()
  while (node !== null) {
    if (node === headingEl) {
      started = true
      skipHeading = true
      node = walker.nextNode()
      continue
    }
    if (node === nextHeadingEl) break
    if (!started) {
      node = walker.nextNode()
      continue
    }
    if (node.nodeType === 1) {
      const el = node as Element
      if (HEADING_TAGS.has(el.tagName)) {
        // A heading inside our ownership window — stop here only if it's the
        // nextHeadingEl (handled above). Otherwise, skip the heading itself
        // but still capture text in its children? No — headings take ownership
        // of their own text, so any heading we encounter other than the next-
        // in-flat-list should not have been produced as a segment. Our caller
        // loops with pairs of consecutive headings, so this branch is dead on
        // the happy path; guard defensively by breaking.
        break
      }
      node = walker.nextNode()
      continue
    }
    // Text node
    if (skipHeading) {
      // Skip text nodes that are descendants of the heading we just saw.
      // headingEl.contains works on happy-dom.
      if (headingEl.contains(node)) {
        node = walker.nextNode()
        continue
      }
      skipHeading = false
    }
    const text = (node.nodeValue ?? '').trim()
    if (text.length > 0) parts.push(text)
    node = walker.nextNode()
  }
  return parts.join(' ').replace(/\s+/g, ' ').trim()
}

function resolveMain(doc: Document): Element {
  const main = doc.querySelector('main#main-content')
  if (main !== null) return main
  return doc.body
}

export function emitPageContent(html: string): PageContent {
  const doc = parseHtml(html)
  const main = resolveMain(doc)

  const headingEls = Array.from(main.querySelectorAll(HEADING_SELECTOR))
  const headings: PageHeading[] = headingEls.map((el) => ({
    level: levelFromTag(el.tagName),
    text: textOf(el),
  }))

  const firstH1 = headings.find((h) => h.level === 1)
  const title = firstH1?.text ?? ''

  const segments: PageSegment[] = []
  for (let i = 0; i < headingEls.length; i++) {
    const headingEl = headingEls[i]
    const nextHeadingEl = i + 1 < headingEls.length ? headingEls[i + 1] : null
    const segmentText = collectSegmentText(headingEl, nextHeadingEl)
    segments.push({
      heading: headings[i],
      text: segmentText,
    })
  }

  return { title, headings, segments }
}
```

Implementation notes:
- Use the literal numeric filter `0x5` (= SHOW_ELEMENT | SHOW_TEXT) because `NodeFilter.SHOW_*` constants may not be available on happy-dom's TreeWalker API — the spec value is stable across DOM implementations.
- `headingEl.contains(node)` — happy-dom Element supports `.contains()`. This guards against the walker visiting text nodes that are CHILDREN of the heading (e.g. the heading's own text); we want to skip them because the heading's own text content belongs to `headings[i]`, not to the segment body.
- `for-of` over NodeListOf is still not safe under ES2022-only lib — but `Array.from(main.querySelectorAll(...))` produces a plain Array which IS iterable with for-of + for-in. The code uses the array form consistently.
- `.replace(/\s+/g, ' ')` collapses interior whitespace for deterministic text comparison. ReDoS safe (single bounded class quantifier).
- `ownerDocument` is used to grab the TreeWalker factory — ensures we stay inside the happy-dom Document produced by parseHtml, NOT the global document.

```typescript
// Fallback: Use heading indices into a flat textContent walk
function collectSegmentText(headingEl: Element, nextHeadingEl: Element | null): string {
  // Get all descendants + own siblings in document order by walking all descendants of body
  // and splitting at heading positions. Simpler but O(n) per call — acceptable for
  // page-sized inputs.
  const root = headingEl.ownerDocument.body
  const allElements = Array.from(root.querySelectorAll('*'))
  const startIdx = allElements.indexOf(headingEl)
  const endIdx = nextHeadingEl === null ? allElements.length : allElements.indexOf(nextHeadingEl)
  const parts: string[] = []
  for (let i = startIdx + 1; i < endIdx; i++) {
    const el = allElements[i]
    if (HEADING_TAGS.has(el.tagName)) continue
    // Only collect leaf elements' text to avoid double-counting
    if (el.children.length === 0) {
      const t = (el.textContent ?? '').trim()
      if (t.length > 0) parts.push(t)
    }
  }
  return parts.join(' ').replace(/\s+/g, ' ').trim()
}
```
